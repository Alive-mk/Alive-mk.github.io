Triton: An Intermediate Language and Compiler for Tiled Neural Network Computations

2026-03-30 6 min read # 实习

该论文致力于解决深度神经网络（DNN）中自定义算子（Custom Operators）开发时面临的生产力与执行效率（Productivity vs. Performance）之间的根本矛盾。

自定义算子：
自定义算子是深度学习框架中允许开发者自行定义的计算单元 (Operator)，本质是遵循框架语义的可调用函数，实现从输入张量→计算逻辑→输出张量的数据流契约，并支持前向传播与反向传播 (自动微分)。

特性 内置算子 自定义算子
灵活性 较低 (固定功能) 极高 (按需定制)
开发难度 无 (开箱即用) 中高 (需编码实现)
性能优化空间 有限 (通用优化) 极大 (针对性优化)
适用场景 通用计算 创新算法、性能瓶颈、领域专用

在 Triton 之前，AI 研究员和底层工程师在实现一种新的、未被标准库支持的神经网络算法（如新型注意力机制或特殊的激活函数）时，面临两难境地：

  • 依赖高度优化的闭源库（如 cuDNN / cuBLAS）：性能极高，但算子固化（Rigidity），无法组合或融合（Operator Fusion）新的操作。例如，一旦引入一个新的数学公式，标准库就无法直接计算，只能退化为原生 PyTorch 分步计算，导致严重的显存带宽浪费（访存瓶颈 Memory-Bound）。

PyTorch 分步计算是指将深度学习模型的前向传播与反向传播过程拆解为多个明确步骤，通过手动控制计算流程实现对中间结果的精细管理、梯度计算的精准控制和性能优化。这是 PyTorch 动态计算图的核心优势，尤其适合科研探索与复杂算法实现PyTorch。

分步计算的本质：通过显式编码定义每一步计算逻辑，替代隐式的整体计算，实现对张量流向与梯度传播的完全掌控。

Memory-Bound（访存受限/带宽瓶颈）：算子的执行时间主要取决于数据从显存搬运到计算单元的时间，而不是实际计算的时间。深度学习中大部分算子（如 Normalization、Activation）都是访存受限的。

Explicit Architecture Exposure（显式架构暴露）：指现有的 GPU 编程语言（如 CUDA）要求程序员直接面对硬件的底层机制。

Shared Memory Allocation & Bank Conflict（共享内存分配与存储体冲突）：GPU 片上有一种极快但容量极小的内存（SRAM）。在 CUDA 中，开发者不仅要手动把数据搬运进 SRAM，还要精心排列数据结构，防止多个线程同时访问同一个存储体（Bank Conflict），否则会导致极严重的性能下降。

  • 使用 CUDA/C++ 手写算子：具备无限的灵活性，但存在极端的显式架构暴露（Explicit Architecture Exposure）。开发者必须手动处理极其复杂的微架构细节。

Triton 提出了一种基于**“分块（Tiled / Block-based）”的编程范式与即时编译（JIT Compiler）架构。它将程序员从线程级（Thread-level）抽象提升到了分块级（Block-level）抽象**。

论文设计了一种新的中间表示语言（Triton IR），并基于 LLVM 构建了编译器。开发者只需要写针对“多维数据块（Tile）”的操作逻辑。Triton 编译器在底层通过一系列编译期优化（Compile-time Optimization Pass），自动将“块级别的操作”映射并拆解为底层 GPU 的“线程级别操作”。

想象你要开设一条极其高效的流水线来做“宫保鸡丁”（开发一个新的 AI 算子）。

  • 以前的做法 1（用现成库，像买冷冻预制菜）：去市场买别人做好的“冷冻宫保鸡丁”（cuDNN），加热一下就能卖，速度很快。但是！如果今天你想加点创意，变成“菠萝宫保鸡丁”，预制菜厂是不提供这个选项的。你只能买一份“宫保鸡丁”和一份“冷冻菠萝”，让员工分别解冻后再拌在一起。这导致来回进出冷库（显存 HBM）极其浪费时间，效率极差。
  • 以前的做法 2（手写 CUDA，像手工指挥 10000 个帮厨）：为了追求极致速度并加入菠萝，你决定自己开工厂。但这家工厂的底层机器（GPU 硬件架构）极其反人类。你作为厂长，不能下达宏观指令，你必须精确指挥 10000 个帮厨（Threads 线程）每个人的左手和右手：
    • “张三，你去 1 号架子拿一块肉，千万别和李四撞在同一个通道里（处理 Bank Conflict）！”
    • “王五，你必须和赵六同时伸手拿相邻的两块肉，这样推车才能一次拉走（手动管理内存合并 Memory Coalescing）！”
    • “大家都停下！等李四把肉放进小盆里（共享内存分配与同步屏障），大家再继续！”
  • 结果：要写出这样的工厂管理手册（手写极速 CUDA 算子），你需要极其顶尖的底层专家（Ninja Programmer），不仅耗时几个月，还很容易写错，导致流水线撞车崩溃。

Triton 论文其实就是发明了一个“超级智能工厂包工头（Triton 编译器）”。

  • 你（开发者）现在只需要怎么做？ 你不需要再管那 10000 个帮厨了。你只需要针对**“一整筐食材（Tile / 分块）”下达指令（也就是块级编程抽象**）。 你只用写一段 Python 代码：“包工头，去冷库拿一筐鸡肉、一筐菠萝放案板上，把这一筐切块，炒熟，然后再把这筐成品端出去。”
  • Triton（智能包工头 / 编译器）在底层做了什么？ 包工头拿到你的要求后，它自己懂硬件微操。它会自动把“一筐食材”的任务完美拆解给那 10000 个帮厨：
    • 它自动计算好帮厨的站位，保证他们拿食材时同时去拿，一车拉走（自动内存合并优化）。
    • 它自动计算案板需要多大，帮厨什么时候需要互相等待（自动分配 SRAM 并做线程同步）。
    • 它甚至会在开工前，先拿几筐不同大小的筐（比如 128个一筐，256个一筐）试炒几次，看看哪个规格在这个型号的厨房里炒得最快，然后固定用这个规格（自动调优 Auto-tuning）。
